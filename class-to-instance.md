# Class → Instance 設計
## 現代魔術における安全な象徴運用のための設計原則

本文書は、神格や象徴をそのまま扱わないために、
オブジェクト指向プログラミング（OOP）の
「クラス／インスタンス」比喩を用いた
安全設計の考え方を示すものである。

クラスとは、構造と属性の定義（型）であり、
インスタンスとは、その型を持つが、
状態（値）は生成時に初期化された個別オブジェクトである。

召喚の文脈では、
クラスは神格・象徴の「型」を指し、
インスタンスは、原典の人格性・履歴・文脈といった
状態を引き継がずに生成された個別実装を指す。

本モデルは、既存の神格や象徴に含まれる
人格・履歴・文脈といった状態を
そのまま再利用することを目的としていない。
扱うのは、クラスそのものではなく、
その時点で生成されたインスタンスを、
生成者自身の責任において運用することである。

## 1. クラスとは何か

ここでいう「クラス」とは、
ある象徴が持ちうる属性（フィールド）と、
それらに結び付けられた
文化的・物語的に共有された初期値を定義した
抽象的な設計単位を指す。

これらの属性には、たとえば以下が含まれる。

- 歴史（神話・設定・過去の物語）
- 意図（象徴に付与された目的や意味）
- 集団的投射（共有されたイメージや期待）
- 美学・恐怖・攻撃性などの強度

クラスは「存在」ではなく、
象徴がどのような属性構造と
初期値を持つと想定されているかを示す
参照用の型（テンプレート）である。

そのため、本モデルにおいて
クラスは直接運用の対象とはならない。

### 例（クラスに該当するもの）

- 神格
- エグレゴア
- シジル体系
- 数（例：663）
- 名前や称号

---

## 2. インスタンスとは何か

インスタンスとは、
クラスで定義された属性構造を参照して生成される
個別のオブジェクトである。

ただし、インスタンスは
クラスに含まれる初期値
（履歴・神話・既存の物語・人格性など）を
そのまま引き継がない。

生成時点において、
インスタンスの値は初期化された状態にあり、
そこから先の状態は
生成者の運用によって形成される。

インスタンスは必ず：

- 作成者が誰か明確であり
- 使用範囲が限定され
- 終了・破棄・再設計が可能

でなければならない。

インスタンスは共有を前提としない。

---

## 3. なぜ分離が必要か

クラスとインスタンスを分離しない場合、
以下の問題が生じる。

- 共有された初期値（神話・履歴・人格性）が
  無批判に再利用される
- 集団的なイメージが
  個別の運用に流入し、制御不能になる
- 実践の結果に対する責任の所在が曖昧になる
- 「正統性」や「再現性」を根拠とした
  権威化が起きる

分離とは、
**初期値をコピーしないための設計**であり、
生成者が責任を引き受けるための
安全装置である。

---

## 4. REBOOT という考え方について

REBOOT とは、

- 既存の初期値を再利用することではなく
- 共有された履歴を再演することでもなく
- 元の象徴を復活・召喚することでもない

クラスを参照点として用いつつ、
初期化された状態から
新しいインスタンスを生成するという
**運用上の宣言**である。

REBOOT は免罪符ではなく、
「このインスタンスの状態と結果は、
生成者自身の責任である」
という境界線を明示するための言葉である。

---

## 5. 禁止事項（重要）

本設計思想において、以下を行わない：

- 原典の召喚手順・儀式の再演  
- クラスを「実在の存在」として扱うこと  
- 不特定多数への実践推奨  
- 集団的エグレゴアの生成・拡散  

これらはすべて、本モデルの目的に反する。

---

## 6. 責任について

インスタンスの作成・使用・終了に関する責任は**すべて作成者本人に帰属する**。クラスは責任を肩代わりしない。

設計と責任の問題である。
